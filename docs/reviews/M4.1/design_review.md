# 1. Workspace Construction Pipeline

Complete execution path from upstream models to returned workspace:

```text
PaperModel
TaskModel
        ↓
WorkflowOrchestrator._run_stage(PipelineStage.CODER, ...)
        ↓
Coder.run(paper: PaperModel, task: TaskModel, patch_plan: PatchPlan | None = None)
        ↓
Coder._paper_slug(paper.title)                    → paper_slug (str)
        ↓
WorkspaceManager.create_workspace(paper_slug)     → Workspace
        ↓
WorkspaceManager.initialize_repository(
    workspace,
    paper.title,
    task,
    patch_plan,
)
        ↓
WorkspaceManager.write_file(workspace, "README.md", ...)
WorkspaceManager.write_file(workspace, "requirements.txt", ...)
        ↓
return Workspace
        ↓
stored in WorkflowHistory.workspace
        ↓
passed to Runner.run(workspace) by WorkflowOrchestrator
```

## Stage-by-stage responsibility

### `PaperModel` and `TaskModel` (inputs)

`PaperModel` supplies the paper title used for slug derivation and README heading. `TaskModel` supplies `paper_title` and `steps` (a list of `TaskStep` objects) used to populate the Engineering Tasks section of `README.md`.

`PatchPlan` is an optional third argument. When present (review retry loop in `WorkflowOrchestrator`), its `analysis` field is appended to `README.md` under a Patch Analysis section.

### `WorkflowOrchestrator`

`WorkflowOrchestrator` schedules `Coder.run()` at the `PipelineStage.CODER` stage. On the first pass it calls:

```python
self._coder.run(history.paper, history.task)
```

On review retry iterations it calls:

```python
self._coder.run(history.paper, history.task, patch_plan)
```

`WorkflowOrchestrator` itself was not modified in M4.1. It continues to store the returned `Workspace` in `WorkflowHistory.workspace` and pass it to `Runner`.

### `Coder.run()`

`Coder` is the agent entry point for workspace construction. It performs orchestration logic only:

1. Derives `paper_slug` from `paper.title` via `Coder._paper_slug()`.
2. Delegates directory creation to `WorkspaceManager.create_workspace(slug)`.
3. Delegates stub file generation to `WorkspaceManager.initialize_repository(...)`.
4. Returns the `Workspace` object produced by `create_workspace`.

`Coder` does not call `Path.mkdir`, `open`, or `write_text` directly. It does not invoke `LLMProvider`. It does not read or write prompt files.

### `WorkspaceManager.create_workspace(paper_slug)`

Creates the on-disk repository root at `{WORKSPACE_ROOT}/{sanitized_slug}/` and creates all required empty subdirectories. Returns a `Workspace` Pydantic model pointing to the created root path.

The slug is sanitized a second time inside `WorkspaceManager._sanitize_slug()` even though `Coder` already produces a slug. This ensures consistent naming regardless of caller.

### `WorkspaceManager.initialize_repository(...)`

Writes the two stub files required by M4.1:

- `README.md` — generated by `_format_readme()`
- `requirements.txt` — generated by `_format_requirements()`

Both files are written through the existing `write_file()` method.

### `Workspace` (output)

A frozen Pydantic model containing:

- `root_path: Path` — absolute or relative path to the workspace root directory
- `paper_slug: str` — sanitized slug string

Downstream agents (`Runner`, and future `Reviewer`/`Coder` retry paths) receive this object.

# 2. Repository Layout

## Root path

Every reproduction workspace is created at:

```text
{WORKSPACE_ROOT}/{paper_slug}/
```

Default `WORKSPACE_ROOT` is `workspace/tasks` (defined in `config.WORKSPACE_ROOT`). The effective path is therefore:

```text
workspace/tasks/{paper_slug}/
```

`paper_slug` is derived from the paper title: converted to lowercase, non-alphanumeric characters replaced with `_`, leading/trailing underscores stripped. If the result is empty, the slug defaults to `"paper"`.

Example: title `"Diffusion Policy: Visuomotor Policy Learning"` → slug `diffusion_policy_visuomotor_policy_learning` → path `workspace/tasks/diffusion_policy_visuomotor_policy_learning/`.

Prior to M4.1, `create_workspace` used a `{timestamp}_{slug}` directory name. M4.1 changed this to `{slug}` only.

## Complete generated structure

```text
workspace/tasks/{paper_slug}/
├── src/
├── configs/
├── scripts/
├── outputs/
├── logs/
├── README.md
└── requirements.txt
```

## Directories

| Directory | Created by | Contents in M4.1 | Responsibility |
|-----------|------------|-------------------|----------------|
| `src/` | `create_workspace()` | Empty | Reserved for future implementation source code |
| `configs/` | `create_workspace()` | Empty | Reserved for future configuration files |
| `scripts/` | `create_workspace()` | Empty | Reserved for future executable scripts |
| `outputs/` | `create_workspace()` | Empty | Reserved for reproduction run outputs within the workspace |
| `logs/` | `create_workspace()` | Empty | Reserved for execution logs within the workspace |

All directories are created via `Path.mkdir(parents=True, exist_ok=True)`. No placeholder files (such as `.gitkeep`) are written inside directories.

## Files

| File | Created by | Responsibility |
|------|------------|----------------|
| `README.md` | `initialize_repository()` via `write_file()` | Documents paper title, project structure, engineering task list, workspace initialization status, and optional patch analysis |
| `requirements.txt` | `initialize_repository()` via `write_file()` | Placeholder dependency file; contains a single comment line |

No other files are generated in M4.1. Specifically, no files are created under `src/`, `configs/`, or `scripts/`.

## `README.md` content structure

Generated by `WorkspaceManager._format_readme()`:

1. **Title** — `# {paper_title}`
2. **Project Structure** — ASCII tree listing all directories and both files
3. **Engineering Tasks** — one bullet per `TaskStep`: `**{id}** ({status}): {name} — {description}`; if `task.steps` is empty, writes `- No tasks defined.`
4. **Workspace Status** — three status lines:
   - `Repository skeleton: created`
   - `Source code generation: not started`
   - `Configuration generation: not started`
   - Followed by: `This workspace is a repository skeleton. Implementation files have not been generated yet.`
5. **Patch Analysis** (conditional) — included only when `patch_plan is not None`; contains `patch_plan.analysis` verbatim

## `requirements.txt` content

Generated by `WorkspaceManager._format_requirements()`:

```text
# Dependencies will be populated during code generation.
```

# 3. WorkspaceManager Design

**Module:** `workspace/manager.py`

`WorkspaceManager` is the only component that performs filesystem operations for repository construction. Agents (including `Coder`) never call `pathlib` write or mkdir APIs directly for workspace creation. This matches the architecture rule stated in `docs/architecture/ARCHITECTURE.md` §7: all file operations are managed by `WorkspaceManager`.

## Module-level constant

| Name | Value | Visibility |
|------|-------|------------|
| `REPOSITORY_SUBDIRS` | `("src", "configs", "scripts", "logs", "outputs")` | Public module constant; imported by tests |

## Public APIs

### `WorkspaceManager.__init__(root: Path | None = None, outputs_dir: Path | None = None) -> None`

**Signature:** `__init__(self, root: Path | None = None, outputs_dir: Path | None = None) -> None`

**Responsibility:** Initialize the manager with workspace root and project-level outputs directory. Creates both directories if they do not exist.

**Parameters:**
- `root` — workspace tasks root; defaults to `config.WORKSPACE_ROOT` (`workspace/tasks`)
- `outputs_dir` — project-level report output directory; defaults to `config.OUTPUTS_DIR` (`outputs`)

**Return value:** `None`

---

### `WorkspaceManager.create_workspace(paper_slug: str) -> Workspace`

**Signature:** `create_workspace(self, paper_slug: str) -> Workspace`

**Responsibility:** Create a new reproduction workspace directory and all required empty subdirectories. This is the entry point for on-disk repository skeleton creation.

**Behavior:**
1. Sanitize `paper_slug` via `_sanitize_slug()`
2. Set `root_path = self._root / slug`
3. Create each subdirectory in `REPOSITORY_SUBDIRS` under `root_path`
4. Return `Workspace(root_path=root_path, paper_slug=slug)`

**Return value:** `Workspace` — frozen Pydantic model with `root_path` and `paper_slug`

**M4.1 change:** Path layout changed from `{timestamp}_{slug}` to `{slug}` only. Public signature unchanged.

---

### `WorkspaceManager.initialize_repository(workspace, paper_title, task, patch_plan=None) -> None`

**Signature:** `initialize_repository(self, workspace: Workspace, paper_title: str, task: TaskModel, patch_plan: PatchPlan | None = None) -> None`

**Responsibility:** Write stub repository files (`README.md` and `requirements.txt`) into an existing workspace. Called after `create_workspace()` completes directory creation.

**Behavior:**
1. Generate README content via `_format_readme(paper_title, task, patch_plan)`
2. Write `README.md` via `write_file()`
3. Generate requirements content via `_format_requirements()`
4. Write `requirements.txt` via `write_file()`

**Return value:** `None`

**Added in M4.1.** This method is new but does not alter any previously frozen method signatures.

---

### `WorkspaceManager.write_file(workspace, relative_path, content) -> None`

**Signature:** `write_file(self, workspace: Workspace, relative_path: str, content: str) -> None`

**Responsibility:** Write a text file at `workspace.root_path / relative_path`. Creates parent directories if needed.

**Return value:** `None`

**Used by:** `initialize_repository()` for `README.md` and `requirements.txt`. Also available for future file writes by other agents via `WorkspaceManager`.

---

### `WorkspaceManager.read_file(workspace, relative_path) -> str`

**Signature:** `read_file(self, workspace: Workspace, relative_path: str) -> str`

**Responsibility:** Read and return the text content of a file within the workspace.

**Return value:** `str` — file content

**Raises:** `FileNotFoundError` if the file does not exist

**Unchanged in M4.1.** Not used by Coder in this milestone.

---

### `WorkspaceManager.write_output(workspace, relative_path, content) -> Path`

**Signature:** `write_output(self, workspace: Workspace, relative_path: str, content: str) -> Path`

**Responsibility:** Write a file into the workspace `outputs/` subdirectory.

**Return value:** `Path` — absolute path to the written file

**Unchanged in M4.1.** Not used by Coder in this milestone.

---

### `WorkspaceManager.write_report(report, filename="report.md") -> Path`

**Signature:** `write_report(self, report: ReportModel, filename: str = "report.md") -> Path`

**Responsibility:** Write the final workflow report to the project-level `outputs/` directory (not inside a reproduction workspace). Formats report content via `_format_report()`.

**Return value:** `Path` — absolute path to the written report file

**Unchanged in M4.1.** Called by `WorkflowOrchestrator` after `Reporter.run()`, not by `Coder`.

---

## Private helpers (not public API)

| Method | Purpose |
|--------|---------|
| `_sanitize_slug(paper_slug)` | Lowercase, replace non-alphanumeric with `_`, strip edges, default `"paper"` |
| `_format_requirements()` | Return placeholder requirements.txt content |
| `_format_readme(paper_title, task, patch_plan)` | Assemble README.md content |
| `_format_report(report)` | Assemble project-level report.md content (pre-existing) |

## Why WorkspaceManager owns filesystem operations

The architecture document requires that agents never manipulate files directly. Centralizing filesystem access in `WorkspaceManager` provides:

- A single location for path conventions (`WORKSPACE_ROOT`, subdirectory names, slug sanitization)
- Consistent parent-directory creation before writes
- Separation between agent orchestration logic (`Coder`) and I/O operations (`WorkspaceManager`)
- Testability: tests can inject a temporary `root` path without modifying agent code

`Coder` holds a reference to `WorkspaceManager` injected at construction time and delegates all repository creation to it.

# 4. Coder Integration

**Module:** `agents/coder.py`

## Constructor

```python
Coder.__init__(
    self,
    workspace_manager: WorkspaceManager,
    llm: LLMProvider | None = None,
) -> None
```

**Dependencies:**
- `workspace_manager: WorkspaceManager` — required; used for all repository construction
- `llm: LLMProvider | None` — optional; stored as `self._llm` but not referenced in M4.1

The `llm` parameter exists in the constructor signature from prior skeleton work. It is retained to preserve the public constructor API. No LLM calls are made in M4.1.

## Execution flow

```python
def run(self, paper, task, patch_plan=None) -> Workspace:
    slug = self._paper_slug(paper.title)
    workspace = self._workspace_manager.create_workspace(slug)
    self._workspace_manager.initialize_repository(
        workspace, paper.title, task, patch_plan
    )
    return workspace
```

Step by step:

1. **`_paper_slug(paper.title)`** — static method; produces slug from paper title using the same regex rule as `WorkspaceManager._sanitize_slug()`.
2. **`create_workspace(slug)`** — creates directory tree; returns `Workspace`.
3. **`initialize_repository(...)`** — writes `README.md` and `requirements.txt`.
4. **Return `workspace`** — same object returned by `create_workspace()`.

## Interaction with WorkspaceManager

`Coder` calls exactly two `WorkspaceManager` methods during M4.1 execution:

| Call | Purpose |
|------|---------|
| `create_workspace(slug)` | Directory skeleton |
| `initialize_repository(workspace, paper.title, task, patch_plan)` | Stub files |

`Coder` does not call `write_file`, `read_file`, `write_output`, or `write_report` directly.

## Returned Workspace object

```python
Workspace(
    root_path=Path("{WORKSPACE_ROOT}/{paper_slug}"),
    paper_slug="{paper_slug}",
)
```

Both fields are set by `WorkspaceManager.create_workspace()`. `Coder` returns this object unchanged.

## What Coder does in M4.1

- Derives workspace slug from paper title
- Delegates repository directory and file creation to `WorkspaceManager`
- Returns `Workspace` to the orchestrator

## What Coder intentionally does NOT do in M4.1

- Does not invoke `LLMProvider.complete()` or any LLM API
- Does not read prompt files or call `PromptBuilder`
- Does not generate Python source files (`src/main.py`, `scripts/train.py`, or any other)
- Does not generate configuration files
- Does not infer or write dependency packages beyond the placeholder comment in `requirements.txt`
- Does not execute code
- Does not call filesystem APIs directly (`mkdir`, `open`, `write_text`)
- Does not modify `Runner`, `Reviewer`, or `Reporter` behavior

## Removed from prior Coder implementation

Before M4.1, `Coder.run()` wrote mock implementation files:

- `src/main.py` — mock entry point with `main()` printing `"Training complete."`
- `scripts/train.py` — import and call `src.main.main()`

These writes were removed. M4.1 replaces them with skeleton-only output.

# 7. Current Limitations

The following boundaries reflect the current M4.1 implementation only.

- **No source code generation.** `src/`, `scripts/`, and `configs/` directories remain empty after `Coder.run()`.
- **No configuration generation.** No files are written under `configs/`.
- **Placeholder requirements only.** `requirements.txt` contains a single comment line; no package names or version pins are written.
- **LLM not used.** `Coder` stores `self._llm` but never calls it.
- **Task dependencies not in README.** `TaskStep` does not include `depends_on`; Planner validates `depends_on` in the extraction dict but does not persist it in `TaskModel`. README lists tasks without dependency relationships.
- **Slug-only workspace path.** Re-running `Coder.run()` for the same paper title targets the same directory. `mkdir(exist_ok=True)` and `write_file()` overwrite existing `README.md` and `requirements.txt` without versioning.
- **No timestamp suffix.** Unlike the pre-M4.1 `create_workspace` behavior (`{timestamp}_{slug}`), concurrent or repeated runs for different papers with identical titles would collide on the same path.
- **Runner mismatch.** `Runner.run()` remains a mock that returns `executed_command="python scripts/train.py"`, but `scripts/train.py` is not created in M4.1.
- **Reviewer retry recreates skeleton only.** When `patch_plan.requires_patch` is `True`, `WorkflowOrchestrator` re-invokes `Coder.run()`, which recreates the skeleton and rewrites stub files but does not apply patches to source files.
- **`initialize_repository` is not in the frozen interface table** in `DEVELOPMENT.md`. It is a new public method added in M4.1.
